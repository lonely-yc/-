# Findings & Decisions

## Requirements
- 使用 `planning-with-files` skill 的文件化方式组织本次复杂文档生成任务。
- 输出一份完整 Markdown，记录从参赛想法到系统成型的主要对话、功能演进、架构设计和问题解决过程。
- 文件必须放到 `E:\HTML\AI赋能参赛\资料` 目录下。
- 文档需要能作为后续复盘、答辩、项目说明和交接材料使用。

## Research Findings
- 项目最终命名为“项目履历平台”。
- 系统核心定位从最初的“开发计划/工时/饱和度工具”逐步演进为“项目履历、需求变更、发包部署、人员交接、资料沉淀”的综合追踪平台。
- 用户多次强调系统要服务于老同事离职、新同事入职、项目交接和现场问题追溯。
- 前端风格参考 Figma 后台管理系统样式，最终采用简洁、清爽、管理台风格。
- 后端采用独立 Spring Boot + MySQL 项目结构，并提供用户、项目、项目成员、履历、文件上传、个人交接资料等接口。

## Technical Decisions
| Decision | Rationale |
|----------|-----------|
| 文档采用阶段化叙事 | 比逐字对话更适合答辩、复盘和后续维护 |
| 把操作日志和项目履历区分说明 | 用户明确要求操作日志记录系统操作，项目履历记录业务事实 |
| 将个人交接资料独立于项目公共资料 | 解决个人理解沉淀和公共项目资料混淆的问题 |
| 保留问题与迭代记录 | 体现系统是通过真实业务反馈逐步成型 |

## Issues Encountered
| Issue | Resolution |
|-------|------------|
| 对话时间跨度长、需求多次变化 | 按业务阶段整理，而不是机械按时间逐条罗列 |
| 多处功能同时包含前后端内容 | 按模块拆分：人员、项目、履历、发包、文件、交接、首页 |
