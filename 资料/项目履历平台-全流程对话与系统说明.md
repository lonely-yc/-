> 生成时间：2026-05-24

## 1. 文档说明

这份文档不是逐字逐句的聊天记录，而是对整个系统从想法产生、需求演进、设计取舍、前后端实现、问题修正到最终成型过程的结构化整理。

它重点回答几个问题：

- 为什么要做这个系统？
- 最初想解决什么问题？
- 需求是如何一步步变化的？
- 系统最终包含哪些功能？
- 前后端架构如何设计？
- 每个模块解决了日常工作中的什么痛点？
- 后续演示或答辩时应该如何讲解？

## 2. 项目背景

本项目来源于“AI赋能·数智创新实践赛”的参赛需求。最初目标是围绕日常工作中的实际问题，设计几个 BS 端小工具，用来解决重复填报、项目投入不清晰、人员饱和度难估算、工作交接困难等问题。

最早的方向包括：

- 开发人员每日工作计划填写工具
- 项目投入工时统计工具
- 人员工作饱和度分析工具
- 项目需求、开发、部署过程追踪工具

随着讨论深入，系统的重心从“工时填报”逐渐转向“项目过程追溯”和“项目交接”。因为在真实工作中，项目最难管理的并不只是每天填了多少工时，而是：

- 需求为什么变了？
- 谁提出的？
- 后端改了什么？
- 前端改了什么？
- 哪次发包解决了什么问题？
- 现场部署用了哪些中间件？
- 配置文件在哪里？
- 新同事接手项目时应该从哪里看？
- 老同事离职后经验如何保留下来？

因此，系统最终定位为：**项目履历平台**。

## 3. 系统定位

项目履历平台不是单纯的项目管理系统，也不是简单的人员管理系统。它更像是一个面向项目交接、问题追溯和过程沉淀的业务履历系统。

一句话定位：

> 用结构化方式记录项目从创建、需求变更、开发改动、发包部署、中间件配置到人员交接的全过程，让项目经验可追溯、可交接、可复用。

系统关注的不是“项目当前是什么状态”这一点，而是“项目一路经历了什么”。

## 4. 初期思考过程

### 4.1 从开发计划填报开始

最初的想法是围绕开发人员每日计划展开。开发人员每天需要填写：

- 投入哪个项目
- 开发内容
- 工作时长
- 计划开始时间
- 计划结束时间
- 完成进度
- 备注

用户提到已有 Excel 字段：

| 字段 | 说明 |
|------|------|
| 序号 | 记录顺序 |
| 项目名称 | 投入项目 |
| 任务模块 | 功能或模块 |
| 详细任务 | 具体开发内容 |
| 工作量(小时) | 投入时长 |
| 开始时间 | 任务开始 |
| 结束时间 | 任务结束 |
| 完成进度 | 当前进展 |
| 备注 | 补充说明 |

最初也讨论过通过这些字段估算人员饱和度和完成星级。

### 4.2 从工时管理转向项目管理

随后用户补充：同一个人员、同一个时间段可以在不同项目之间重叠投入，因此不能简单按时间段排班，而要根据项目工时、工作内容、完成时间综合估算。

这时系统开始从“工时统计”向“项目管理和履历追踪”转变。

### 4.3 从项目管理转向项目履历

进一步讨论后，用户提出更关键的需求：

- 不同职位权限和角色
- 项目经理、开发经理、前端、后端、实施、设计、测试等人员
- 每个人可以填写任务
- 首页日历展示排班和项目工时
- 项目看板展示涉及项目、人员、工时、进度
- 项目从需求、开发到部署都有流水线记录
- 需求变更要记录变了什么
- 后端代码改了什么
- 前端页面改了什么
- 谁提出的客户要求
- 实施发包部署记录
- 中间件配置、现场 IP、端口、配置文件
- 发包频率、改动率统计

此时系统核心逐渐确定为“项目履历和过程追踪”。

- 比“项目管理”更突出本系统特色。

## 6. 用户痛点总结

系统主要解决以下真实工作问题。

### 6.1 项目过程靠人记

很多项目的关键信息散落在聊天记录、邮件、Excel、个人电脑或口头沟通中。项目一旦周期变长，或者人员变动，就很难知道当时为什么这么做。

### 6.2 需求变更不可追溯

需求变更常见问题：

- 谁提出的？
- 为什么变？
- 改了哪些业务？
- 后端改了哪些接口？
- 前端改了哪些页面？
- 数据库有没有改表？
- 最终哪次发包解决？

如果没有结构化记录，新同事只能靠问人。

### 6.3 发包部署信息不完整

发包部署常见问题：

- 版本号是什么？
- 发到了测试还是正式？
- 发包文件在哪里？
- 本次解决了哪些问题？
- 是否有回退方案？
- 配置文件是否同步变更？

系统需要把发包记录和需求变更关联起来。

### 6.4 中间件配置难交接

项目现场可能使用：

- MySQL
- Redis
- MQ
- Nginx
- ElasticSearch
- MinIO
- 其他中间件

需要记录：

- 环境：测试 / 正式
- 类型
- 地址
- 端口
- 配置文件路径
- 配置文件上传
- 说明

并支持查看和下载配置文件。

### 6.5 人员交接资料缺失

老同事离职、新同事入职时，公共项目资料并不能完全解决交接问题。每个人对项目都有自己的理解资料，比如：

- 自己负责的模块说明
- 接口调用经验
- 现场问题处理记录
- 数据来源说明
- 申请流程说明
- 对接人信息
- 个人踩坑总结

因此系统新增了“个人项目交接资料”，并与项目公共资料隔离。

## 9. 首页设计

首页用于管理者快速了解整体情况。

### 9.1 区域简况

首页展示关键统计：

- 项目总数
- 参与人员
- 需求变更数量
- 发包记录数量
- 项目成员投入人次
- 中间件配置数量
- 风险预警数量

这些指标不是为了炫技，而是帮助项目经理快速判断当前项目风险和维护压力。

### 9.2 项目展板

项目展板按照修改次数、发包次数、中间件完整度等指标排序，优先展示风险较高或变更较频繁的项目。

每张项目卡片包含：

- 项目名称
- 风险原因
- 修改次数
- 发包次数
- 中间件数量
- 风险等级
- 查看入口

### 9.3 风险预警

风险预警关注：

- 需求和发包是否过于频繁
- 中间件配置是否缺失
- 部署信息是否完整
- 项目是否存在未闭环问题

### 9.4 操作日志

操作日志记录系统行为，而不是业务履历。

例如：

- 谁新增了项目
- 谁修改了项目
- 谁删除了资料
- 谁添加了成员
- 谁新增了需求
- 谁新增了发包记录

操作日志和项目履历是分开的：

- 操作日志：记录系统操作。
- 项目履历：记录项目业务事实。

## 12. 项目履历设计

项目履历是系统最核心的设计。

### 12.1 履历目标

项目履历要回答：

- 项目什么时候创建？
- 谁加入了项目？
- 什么时候发生了需求变更？
- 改了什么？
- 什么时候发包？
- 发包解决了哪些问题？
- 中间件什么时候变更？
- 现场问题如何处理？
- 项目最终如何归档？

它的目标不是做一个普通时间轴，而是帮助后续接手人员理解项目脉络。

### 12.2 时间轴视图

时间轴分为两种：

1. 项目卡片预览
2. 详情页项目履历

项目卡片预览：

- 展示最近 5 条关键事件
- 默认按业务发生时间倒序
- 用于快速看最近发生了什么

详情页项目履历：

- 展示完整时间轴
- 默认从项目创建到当前正序
- 可切换最新优先
- 适合新人按过程阅读

### 12.3 事件类型

项目履历事件类型包括：

| 类型 | 说明 | 建议颜色 |
|------|------|----------|
| 项目创建 | 立项、客户、负责人、初始范围 | 青绿色 |
| 人员加入 | 谁加入、负责什么职责 | 绿色 |
| 需求变更 | 谁提出、为什么改、改了哪些业务 | 橙色 |
| 开发改动 | 后端、前端、数据库改动 | 蓝灰色 |
| 发包部署 | 版本、环境、解决问题 | 蓝色 |
| 中间件变更 | MySQL、Redis、Nginx 等配置变化 | 紫色 |
| 问题修复 | 现场问题、原因、处理方式 | 红色 |
| 项目归档 | 最终版本、资料、遗留问题 | 深灰色 |

### 12.4 时间轴排序原则

项目履历按“业务发生时间”排序，而不是按录入时间排序。

如果补录历史记录：

- 时间轴位置按业务发生时间放置
- 卡片角落显示补录时间，避免误解

同一天多条事件：

- 归并在同一个日期下面
- 日期只出现一次
- 事件按录入时间或业务时间细分排序

### 12.5 时间轴卡片内容

每条履历卡片建议包含：

- 时间
- 类型标签
- 标题
- 发起人 / 负责人
- 变更原因
- 改动内容
- 后端改动
- 前端改动
- 数据库改动
- 中间件改动
- 关联资料
- 影响范围
- 状态

## 13. 人员管理模块

人员管理最初只是人员基础信息维护，后来扩展成“人员项目交接入口”。

### 13.1 人员列表

人员列表字段：

- 姓名
- 工号
- 账号
- 手机号
- 岗位
- 所属项目
- 操作

支持：

- 新增人员
- 编辑人员
- 删除人员
- 分页
- 所属项目标签展示

### 13.2 所属项目标签

每个人员可以关联多个项目，在人员列表中以标签形式展示所属项目。

这样可以快速知道：

- 这个人参与了哪些项目
- 是否有多个项目同时投入
- 后续交接需要覆盖哪些项目

### 13.3 个人项目交接展板

点击人员后，在人员表格下方展示该成员参与项目的交接概览。

展板采用紧凑卡片网格风格，参考 Figma 里的后台监控卡片：

- 白底
- 细边框
- 轻阴影
- 小圆角
- 信息密度适中

每张项目卡展示：

- 项目名称
- 项目编号
- 客户
- 负责人
- 项目状态
- 当前人员职责
- 加入时间
- 个人资料数量
- 最近上传时间
- 交接状态
- 交接人
- 操作：详情、上传

### 13.4 个人项目资料

个人项目资料与项目公共资料完全分开。

个人资料用于记录：

- 个人对项目的理解
- 接手说明
- 模块说明
- 接口说明
- 现场问题经验
- 个人整理的文档

文件保存路径：

`uploads/person-handover/{personId}/{projectId}/`

这样可以保证资料按人员和项目隔离。

### 13.5 交接信息

交接信息包含：

- 交接人
- 交接状态
- 交接时间
- 交接说明

交接状态建议：

- 未交接
- 交接中
- 已交接
- 需补充

- 谁支撑
- 现场遇到什么问题

因此在项目详情的相关人员区域增加“项目过程备注”。

备注类型包括：

- 接口对接
- 数据来源
- 支撑协调
- 问题记录
- 其他事项

用户后续要求去掉“数据中台”这一单独下拉选项，因为数据中台相关内容可以归到数据来源、接口对接或支撑协调中。

过程备注字段包括：

- 备注类型
- 发生时间
- 标题
- 联系人 / 支撑人
- 联系电话
- 地址 / 接口
- 申请方式
- 文档 / 申请单名称
- 文档 / 图片地址
- 状态
- 事项说明

这个模块服务于项目接手时最容易丢失的“隐性经验”。

## 21. 演示讲解建议

答辩或演示时可以按以下顺序讲：

### 21.1 讲背景

“我们在实际项目中遇到一个很典型的问题：项目做久了之后，很多信息靠人记。老同事离职、新同事接手时，很难知道需求为什么变、代码为什么改、哪次发包解决了什么问题、现场配置在哪里。”

### 21.2 讲目标

“所以我们设计了项目履历平台，把项目从创建到交付的全过程结构化沉淀下来。”

### 21.3 讲核心能力

“系统围绕三个核心对象展开：项目、人员、履历。项目记录业务过程，人员记录参与和交接，履历记录每次关键事件。”

### 21.4 讲项目履历

“履历不是普通日志。它记录业务事实，比如需求变更、发包部署、中间件变化、人员加入和现场问题。新同事可以像读项目简史一样理解项目。”

### 21.5 讲交接价值

“个人项目资料和公共项目资料分开，解决了个人理解无法沉淀的问题。每个人可以上传自己对项目的理解资料，并设置交接人和交接说明。”

### 21.6 讲技术实现

“前端采用 React + Vite，后端采用 Spring Boot + MyBatis + MySQL，文件先存储到本地 uploads 目录，后续可以平滑迁移到服务器或对象存储。”

### 21.7 讲实际价值

“这个系统不是为了替代项目管理，而是补上项目过程沉淀和交接追溯这一块，让项目经验不再只留在人的脑子里。”

## 23. 总结

项目履历平台的核心价值是：

> 把项目中最容易丢失的过程信息沉淀下来，让项目能被追溯，让经验能被交接，让后来的人更容易接手。

它从最初的开发计划填报工具，逐渐演进为一个围绕项目履历和人员交接的完整平台。这个演进过程本身也说明，真正有价值的系统往往不是一次性设计出来的，而是在真实业务反馈中逐步长出来的。

当前版本已经能够支撑演示、答辩和后续迭代。
